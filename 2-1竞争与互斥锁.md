

多线程的竞争
```
两个线程操作同一公共资源时，一个线程的操作可能会破坏另一个线程操作的完整性，就会造成数据污染或逻辑错误
并发状态下，一个线程可能还没操作完，核心就跳转另一个线程并修改了公共资源，原线程再操作时数据就是错误的
并行状态下，一个线程可能还没操作完，公共资源就被另一个线程修改了，此时原线程再操作所需数据时就是错误的

所以多线程中是否会产生竞争，取决于线程操作和数据的原子性(即完整和连续性)是否被破坏
```

```
int g_count=0;

void test1()
{
	for(int i=0;i<50000;i++)
		g_count++;
}

int main()
{
	thread t1(test1);
	thread t2(test1);
	cout<<g_count<<endl;
	
	return 0;
}

两个线程分别对一个全局变量累加5万次，则最后期望结果应该是10万，但实际会因为竞争的问题导致部分操作被重复
分析：g_count++是非原子操作，在底层分为三步，寄存器读取g_count的值，寄存器值+1，寄存器值写回g_count
两个线程有可能会同时读取到共享变量的旧值，所以他们的部分操作是重复的(若累加过小会因为速度问题体现不明显)
```

原语与原子操作
```
原子操作是一种不可分割的最小操作行为，一旦执行要么成功要么失败，中途不能被打断，而primitive
原语就是计算机中提供的原子操作(其原子性由硬件保证)，是一组用于完成某一功能的程序指令集，调用
某一原语时，必须等待该原语执行完，其他相同原语才能执行，所以我们可通过原语来保证线程的原子性
```

互斥锁(mutex lock)
```
锁也是一种共享的资源，由一对原语加锁和解锁共同构成，其主要作用
有两个：1.保证线程的原子性 2.改变线程状态提高核心资源的利用率

保证线程原子性: 
加锁和解锁之间的区域称为临界区，每一个不同的互斥锁都是一对不同的原语，多个线程使用相同的锁时(加锁)
只有一个能加锁成功(称为持有)，持有锁的线程才能进入临界区，并且临界区执行完再解锁后其他原语才能加锁
所以在临界区中存储的数据和进行的操作能够保证其原子性，对于解锁必须由持有者来进行，否则可能造成死锁

提高核心资源利用率：
每个互斥锁内部都拥有并自动管理一个独立的等待队列，通过某一互斥锁而挂起的线程会存储在其对应的等待队列中
加锁时如果锁空闲则线程状态不变，存放在就绪队列继续等待核心资源的分配，若被占有线程则从运行态变为阻塞态
它会退出就绪队列进入等待队列，不再占有核心资源的分配，直到解锁OS再从等待队列将其唤醒并重新进入就绪队列
```

锁的粒度选择
```
即锁的范围，根据场景选择不同的范围，提示:RAII锁的范围可以通过{}定义局部区来限定
粗粒度锁：范围大，阻塞时间长，并发性低但开销小(加解锁次数)，适用于大段代码的保护
细粒度锁：范围小，阻塞时间短，并发性高但开销大(加解锁次数)，适用于小段代码的保护
```

```
std::mutex lock_1;
std::mutex lock_2;

//保护整个线程的原子性，并发低开销小(1次加解锁，阻塞时长为整个循环)
void test1(){
	lock_1.lock();
	for(int i=0;i<50000;i++)
		g_count++;
	lock_1.unlock();
}

//保护自增操作和数据的原子性，并发高开销大(50000次加解锁，阻塞时长为单语句)
void test2(){
	for(int i=0;i<50000;i++){
		lock_2.lock();
		g_count++;
		lock_2.unlock();
	}
}
```

手动加锁的风险
```
未解锁造会成相关线程阻塞，若该线程销毁则会永久阻塞

1.线程加锁后忘记解锁 
2.异常或提前返回导致未解锁
```

RAII互斥锁
```
采用资源获取即初始化的核心思维，将锁的生命周期与对象进行绑定，由对象管理锁
C++中提供了两个RAII模板类互斥锁，lock_guard和unique_lock，头文件mutex
是mutex类的专属模板类，对mutex类进行了二次封装，专门用来对mutex锁进行管理
```

lock_guard守卫锁
```
特点:
1.锁不可复制、移动
2.极简、安全，性能开销低
3.mutex的专属模板类
```

| 成员方法  | 参数列表                  | 备注                   |
| ----- | --------------------- | -------------------- |
| 有参构造1 | mutex mtx             | 构造并绑定mutex锁，然后加锁     |
| 有参构造2 | mutex  mtx,adopt_lock | 接管mutex锁(默认已持有,不再加锁) |
| 析构    | 无                     | 析构并解锁                |

unique_lock独占锁
```
特点:
1.可移动、手动加解锁，不可复制
2.灵活，但性能开销略高
3.mutex的专属模板类
```

| 成员方法     | 参数列表                  | 备注                                |
| -------- | --------------------- | --------------------------------- |
| 无参构造     | 无                     | 构造但不绑定mutex锁(后续可通过移动赋值绑定)         |
| 有参构造1    | mutex mtx             | 构造并绑定mutex锁，然后加锁                  |
| 有参构造2    | mutex mtx,adopt_lock  | 接管mutex锁(默认已持有,不再加锁)              |
| 有参构造3    | mutex mtx,defer_lock  | 构造并绑定mutex锁，延迟加锁(不再加锁，需后续手动加锁)    |
| 有参构造4    | mutex mtx,try_to_lock | 构造并绑定mutex锁，然后尝试加锁，成功持有，失败返回(不阻塞) |
| 析构       | 无                     | 析构并解锁                             |
| lock     | 无                     | 加锁                                |
| unlock   | 无                     | 解锁                                |
| try_lock | 无                     | 尝试加锁，成功持有，失败返回(不阻塞)               |

复制与移动
```
复制：通过拷贝构造或=创建锁对象的副本(存在两个对象都认为自己占有锁，会导致重复解锁以及竞争的问题)
移动：通过std::move方法或=，将mutex锁的绑定从一个对象转移到另一个对象(转移后原对象不再绑定该锁)

unique_lock<mutex> u1,u2;       //构造但不绑定
unique_lock<mutex> u3(m3);      //构造并绑定m3，然后加锁
u1=unique_lock<mutex>(m1);      //构造匿名对象并绑定m1，然后加锁，再将匿名对象转正给u1
u2=move(u3);                    //将锁的绑定从u3转移到u2
```

```
#include<iostream>
#include<thread>
#include<mutex>
using namespace std;
  
int g_count=0;
mutex m1;
  
void test1()
{
    for(int i=0;i<50000;i++)                            
    {                                                  
        lock_guard<mutex> guard1(m1);
        //lock_guard<mutex> new_guard=move(guard1);   无法移动
        //lock_guard<mutex> new_guard(guard1);        无法拷贝构造
        //lock_guard<mutex> new_guard=guard1;         无法等号赋值
        g_count++;
    }
}

void test2()
{
    for(int i=0;i<30000;i++)
    {
        unique_lock<mutex> unique1(m1,defer_lock);    //延迟加锁
        unique1.lock();                               //手动加锁
        unique_lock<mutex> new_unique=move(unique1);  //转移绑定关系,此时已加锁,新对象只需负责解锁
        //unique_lock<mutex> new_unique=unique1;      无法拷贝构造
        //unique_lock<mutex> new_unique(unique1);     无法等号赋值
        g_count++;
        new_unique.unlock();
    }
}

void test3()
{
    for(int i=0;i<20000;i++)
    {
        m1.lock();                                    //手动加锁
        g_count++;
        m1.unlock();                                  //手动解锁
    }
}

int main()
{
    thread t1(test1);
    thread t2(test2);
    thread t3(test3);
    t1.join();
    t2.join();
    t3.join();
    
    cout<<"result:"<<g_count<<endl;
    return 0;
}
```

锁的注意事项
```
1.对于不同的场景要选择合适的锁、以及合理的锁粒度
2.加锁和解锁要调用对应的锁方法，例如用独占锁，解锁则应调独占锁的lock方法
3.注意不要重复加解同一个锁，重复加锁可能会导致单线程锁死，重复解锁导致竞争

提示：独占锁手动解锁不会导致重复(但只能有一次)，析构时会检测以避免重复解锁
```
